# LEVERAGE FRAMEWORK
## Ronin Operating System v1.0 — How to Measure and Improve ROS Success
**Established:** 2026-06-26
**Status:** Active — governs how ROS performance is evaluated

---

## THE ROS PRINCIPLE

> Do not measure success by the number of agents.
> Measure success by leverage.

Leverage is the ratio of output to input. A system with high leverage produces significantly more output — revenue, reach, impact, knowledge — per unit of Creator time, attention, and energy than a system without it.

The question is never "how many components does ROS have?"
The question is always: **"Does ROS make the Creator more capable?"**

---

## THE FIVE LEVERAGE QUESTIONS

Every quarter, evaluate ROS against all five:

| Question | What a YES looks like | What a NO looks like |
|----------|----------------------|---------------------|
| **Does ROS help the Creator create more?** | More transmissions published per week than before ROS | Creator still manually managing every publication step |
| **Does ROS help the Creator serve more?** | More clients served without proportionally more time | Client load is still 1:1 with Creator hours |
| **Does ROS help the Creator earn more?** | Revenue growing faster than Creator time invested | Revenue requires constant manual intervention to maintain |
| **Does ROS help the Creator teach more?** | Knowledge is being preserved, shared, and found | Doctrine lives only in the Creator's memory |
| **Does ROS help the Creator remember more?** | Past decisions, lore, and relationships are retrievable | Creator re-explains context at the start of every session |

**If all five are YES:** ROS is working. Expand it.
**If any is NO:** Fix that dimension before expanding others.

## THE LEVERAGE REVIEW

**Cadence:** Every 90 days (end of each quarter)

**Process:**
1. Pull current metrics across all five dimensions
2. Compare to targets
3. Identify the one dimension with the lowest leverage score
4. Recommend one ROS improvement that would most increase that dimension's leverage
5. Creator approves or redirects the recommendation
6. Improvement becomes the primary non-revenue sprint objective for the next 30 days

**Format:**
```
LEVERAGE REVIEW — [Quarter] [Year]

REVENUE LEVERAGE:    [Score: 0–10] — [Trend: UP/FLAT/DOWN]
CREATION LEVERAGE:   [Score: 0–10]
SERVICE LEVERAGE:    [Score: 0–10]
KNOWLEDGE LEVERAGE:  [Score: 0–10]
TEACHING LEVERAGE:   [Score: 0–10]

LOWEST LEVERAGE DIMENSION: [Name]
ROOT CAUSE: [Why it's low]
RECOMMENDED FIX: [Specific improvement]
COST: [Sprint time or resources required]
REVENUE IMPACT: [Expected effect on revenue]
AWAITING CREATOR APPROVAL: [YES / NO]
```

## THE LEVERAGE TRAP

The most common failure in organizational systems is **complexity for its own sake**. Systems that require maintenance become burdens. Burdens reduce leverage. The ROS Principle exists to prevent this.

**The Leverage Trap signals:**
- More time is spent maintaining ROS than benefiting from it
- Agents produce reports that no one reads
- Processes are followed because they exist, not because they help
- New components are added to solve problems that simpler approaches could solve

**When a Leverage Trap is detected:**
1. Pause expansion
2. Audit the component causing the drag
3. Simplify or remove it
4. Document what was learned

> If removing a component of ROS makes the Creator more capable, that component should not exist.
